KONSEP 1/3

TRUNK-BASED VS
GITFLOW

Branching strategy menentukan seberapa cepat kode sampai ke production. Untuk tim modern dengan CI/CD yang kuat, trunk-based hampir selalu menang. GitFlow masih cocok untuk rilis berjadwal dengan multiple versi yang di-maintain.

✅ TRUNK-BASED
# short-lived feature branch
git checkout -b feat/payment
# commit kecil, push sering
git push origin feat/payment
# merge ke main < 24 jam
gh pr create --base main
  • • Branch umur pendek (< 1 hari)
  • • Feature flag untuk work-in-progress
  • • Cocok untuk continuous deployment
⚙️ GITFLOW
# long-lived branches
main     # production
develop  # integration
feature/* release/* hotfix/*
  • • Multiple versi di-support paralel
  • • Release cadence > 1 minggu
  • • Overhead merge lebih tinggi
📝 KEPUTUSAN PRAKTIS

Default pilih trunk-based. Pindah ke GitFlow hanya jika kamu benar-benar perlu maintain 2+ versi production secara bersamaan (library, SDK, atau enterprise software dengan LTS).

KONSEP 2/3

INTERACTIVE REBASE:
HISTORY TIDY-UP

Sebelum buka PR, rapikan commit history. 15 commit WIP yang berantakan → 3 commit yang bermakna. Ingat: jangan rebase branch yang sudah di-share.

COMMAND
# rebase 5 commit terakhir interaktif
git rebase -i HEAD~5

# editor terbuka dengan daftar commit:
pick a1b2c3d feat: add payment form
pick e4f5g6h fix typo          # → ubah jadi "fixup"
pick i7j8k9l wip debug         # → "drop"
pick m1n2o3p feat: stripe int  # → "reword"
pick q4r5s6t fix: validation   # → "squash"
PICK
Gunakan commit apa adanya
SQUASH / FIXUP
Gabung ke commit di atasnya (squash = keep message, fixup = drop)
REWORD
Ubah pesan commit saja
DROP
Buang commit (hapus baris)
KONSEP 3/3

CONVENTIONAL
COMMITS

Format terstruktur type(scope): subject yang memungkinkan changelog dan semver otomatis. Disiplin format ini langsung terasa manfaatnya di minggu kedua.

❌ BURUK
fix stuff
update
wip
final version
asdfg
changes

Tidak ada konteks. Tidak bisa generate changelog. Tidak bisa auto-bump version.

✅ BAIK
feat(auth): add SSO via Google OIDC
fix(payment): handle 3DS challenge timeout
docs(api): add /v2/orders examples
refactor(db): extract repository layer
test(order): cover refund edge cases
chore(deps): bump vitest to 1.4.0
📝 TYPE REFERENCE
  • • feat — fitur baru (minor bump)
  • • fix — bug fix (patch bump)
  • • docs — dokumentasi saja
  • • refactor — restructure tanpa perubahan perilaku
  • • test — tambah/ubah test
  • • chore — tooling, deps, CI
  • • feat! atau BREAKING CHANGE: → major bump
SIDE-BY-SIDE

MERGE CONFLICT:
PANIK VS TENANG

❌ PANIK
# rollback semua pekerjaan
git reset --hard origin/main
# atau worse...
rm -rf .git
git clone <repo>
  • • Kehilangan commit lokal
  • • Kehilangan stash
  • • Tim jadi tidak percaya
✅ TENANG
# 1. pull latest main
git fetch origin
git rebase origin/main

# 2. baca marker <<< === >>>
# 3. resolve manually atau:
git checkout --ours  file.ts
git checkout --theirs file.ts

# 4. mark resolved
git add file.ts
git rebase --continue
  • • git status setiap langkah
  • • git rebase --abort selalu aman
  • • Pakai mergetool (VS Code, Beyond Compare)
AI-ASSISTED WORKFLOW

GIT WORKFLOW
DENGAN AI TOOLS

AI coding agents bisa drastically mempercepat commit discipline dan PR hygiene. Berikut tiga workflow harian yang proven.

✍️

Claude Code — /commit Draft Message

CLAUDE CODE

Setelah staging file, panggil skill /commit di Claude Code. Skill ini baca diff dan hasilkan conventional commit message:

# Claude Code session
/commit

# Claude akan:
# 1. jalankan git status + git diff --staged
# 2. analisa perubahan
# 3. draft commit dengan format:
#    feat(auth): add SSO via Google OIDC
#
#    - add oidc-client dependency
#    - wire callback route
#    - update session model
🔀

Aider — Git-Native Commit-per-Change Refactor

AIDER

Aider auto-commit setiap kali edit file. Cocok untuk refactor bertahap — setiap step jadi commit yang bisa di-revert independen:

# start aider di repo root
aider src/services/payment.ts

# Aider prompt
Refactor processPayment into 3 functions:
validateCard, chargeCard, recordTransaction.
Commit each extraction as separate commit
with conventional commit message.
📋

GitHub Copilot — PR Description Generation

COPILOT

Di halaman "New Pull Request" GitHub, klik ikon Copilot untuk generate body dari commit history:

# Copilot Chat prompt di PR page
Generate a PR description from the commits in this branch.
Structure: Summary (3 bullets), Test Plan (checklist),
Breaking Changes (if any). Follow our team template.
⚠️ Pitfall: AI-generated commit messages sering terlalu generik ("update files"). Selalu review dan edit untuk menangkap why, bukan hanya what. Gunakan AI sebagai first draft, bukan final answer.
QUIZ • VALIDATE

TEST
PEMAHAMAN

Kamu sudah push feature branch ke remote dan sudah ada 2 reviewer yang comment. Kamu sadar ada typo di commit message paling awal. Apa yang aman kamu lakukan?

A. Langsung git rebase -i lalu git push --force
B. Tambah commit baru fix: typo, biar history apa adanya sampai squash-merge
C. git reset --hard ke HEAD~5 dan commit ulang semua
D. Hapus branch dan bikin baru dari scratch

Jawabannya B — branch sudah di-share dan sudah di-review, sehingga rebase/force-push akan me-reset state reviewer (comment bisa orphan, diff jadi membingungkan). Solusi aman: tambah commit fix biasa, lalu squash-merge saat PR merged akan bikin history akhirnya bersih di main.
