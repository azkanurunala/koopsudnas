KONSEP 2/3

REVIEW TONE:
TAG YOUR INTENT

Comment tanpa tag bikin author bingung: "apakah ini harus di-fix atau cuma saran?". Prefix komentar dengan tag explicit supaya psychological safety terjaga dan author tahu apa yang blocker.

blocking:

Harus di-fix sebelum merge. Security, correctness, data loss.

blocking: SQL query ini vulnerable ke injection.
Pakai parameterized query atau Prisma.
suggestion:

Alternatif yang lebih baik, tapi author boleh menolak dengan alasan.

suggestion: Kayanya lebih bersih kalau extract
validateInput ke helper terpisah. Tapi kalau kamu
punya alasan inline, oke juga.
nit:

Kosmetik kecil (typo, spacing). Author boleh ignore kalau mau.

nit: typo "recieve" → "receive"
question:

Minta penjelasan. Bukan tuntutan perubahan.

question: Kenapa pakai setTimeout di sini
instead of await? Apakah ada issue dengan async?
KONSEP 3/3

ASYNC-FIRST
WORKFLOW

Meeting adalah last resort, bukan default. Tim modern mengerjakan 80% komunikasi secara async supaya orang bisa fokus dan timezone tidak jadi masalah.

ESKALASI CHANNEL
  1. Code comment / PR — default
  2. Async doc (ADR) — decision perlu konteks
  3. Chat (Slack) — koordinasi ringan
  4. Async video (Loom) — demo/walkthrough
  5. Sync meeting — kalau 4 level di atas gagal
ADR TEMPLATE
# ADR-012: Pilih PostgreSQL

## Status
Accepted — 2026-03-15

## Context
Kita butuh relational DB untuk
order + inventory dengan ACID.

## Decision
Gunakan PostgreSQL 16.

## Consequences
+ ACID guarantees
+ JSON columns untuk flexibility
− Ops overhead vs managed Firestore
📝 PAIR VS MOB VS SOLO
  • • Solo — well-understood task, dokumentasi, refactor kecil
  • • Pair — design decision, onboarding, tricky debugging (driver-navigator, swap 25 menit)
  • • Mob — architectural decision, kickoff fitur besar, knowledge transfer (3-5 orang, driver rotate 10 menit)
SIDE-BY-SIDE

REVIEW COMMENT:
TONE MATTERS

❌ TOXIC
"This is wrong."
"Why would you do this???"
"Rewrite this."
"LOL what is this?"
"Junior mistake."
  • • Attack person, bukan code
  • • Tidak ada alasan
  • • Tidak ada solusi
  • • Destroy team trust
✅ CONSTRUCTIVE
blocking: this causes a race condition
when two requests arrive simultaneously.
We saw this in incident #421. Consider
using a transaction with SELECT FOR UPDATE.

suggestion: extract these 3 branches into
a strategy map for readability. Not a must.
  • • Attack code, bukan person
  • • Jelaskan why
  • • Tawarkan solusi konkret
  • • Tag level (blocking vs suggestion)
AI-ASSISTED WORKFLOW

CODE REVIEW
DENGAN AI TOOLS

AI first-pass review menghilangkan trivial feedback loop (lint, typo, obvious bugs). Human reviewer bisa fokus ke arsitektur dan business logic.

🐰

CodeRabbit / Copilot Review — First-Pass Auto Review

CODERABBIT

Install CodeRabbit atau enable Copilot Review di repo. Setiap PR otomatis dapat summary + inline comment untuk issue umum:

# .coderabbit.yaml
reviews:
  profile: chill
  request_changes_workflow: false
  high_level_summary: true
  auto_review:
    enabled: true
    drafts: false
language: id-ID
🔍

Claude Code — /review-pr Skill

CLAUDE CODE

Sebelum request review dari teammate, self-review pakai Claude Code. Skill /review-pr simulasikan review experience:

# Claude Code session
/review-pr 1234

# Claude akan:
# 1. gh pr view 1234 --json + diff
# 2. analisa perubahan dari perspektif reviewer
# 3. flag: security, perf, correctness, tests
# 4. tag comment: blocking / suggestion / nit
# 5. tulis summary yang bisa di-paste ke PR
💡

Cursor Chat — Explain Diff to Reviewer

CURSOR

Saat PR kamu kompleks, pakai Cursor Chat untuk generate walkthrough yang bisa ditempel di PR description:

# Cursor Chat (Cmd+L)
@diff main...HEAD

Generate a reviewer walkthrough in Bahasa Indonesia:
1. Urutan file yang sebaiknya dibaca dulu
2. Key design decisions + alasan
3. Edge cases yang sudah di-cover test
4. Yang sengaja di-defer (TODO)
⚠️ Pitfall: AI review BUKAN pengganti human review. AI bagus catch style/typo/obvious bug, tapi jelek di domain logic, business rules, dan team conventions. Jangan merge hanya berdasarkan AI approval.
QUIZ • VALIDATE

TEST
PEMAHAMAN

Kamu review PR yang menggunakan pendekatan berbeda dari style kamu, tapi tetap benar secara fungsional. Bagaimana cara memberi komentar yang sehat?

A. Tandai blocking: dan minta author rewrite sesuai style kamu
B. Tidak usah comment, biar tidak ribet
C. Tandai suggestion: dengan alasan, lalu approve — biar author yang putuskan
D. Minta meeting 30 menit untuk diskusi

Jawabannya C — kode benar secara fungsional, jadi bukan blocker. Tag suggestion: memberi author autonomy untuk menolak dengan alasan. Review tidak boleh jadi ajang imposisi preferensi personal.
