KONSEP 1/3

CAP THEOREM:
PILIH DUA

Dalam distributed system, kamu tidak bisa punya ketiganya: Consistency, Availability, Partition tolerance. Karena network partition (P) adalah fakta kehidupan, pilihan sebenarnya adalah CP vs AP.

🔒

CP

Konsisten tapi bisa refuse request saat partisi.

Contoh: PostgreSQL, MongoDB (majority), Etcd, Zookeeper

🌐

AP

Selalu available, eventual consistent.

Contoh: Cassandra, DynamoDB, Redis (cluster)

⚡

CA

Hanya di single-node DB (tidak ada partisi).

Contoh: SQLite, single Postgres

📝 KEPUTUSAN PRAKTIS
  • • Finance / booking / inventory → CP (tidak boleh double-charge)
  • • Social feed / product catalog / analytics → AP (oke lihat data lama)
  • • Session / cache → AP (reconstruct dari source)
KONSEP 2/3

HORIZONTAL
SCALING

Scale up (vertical) dulu sampai 1 mesin besar tidak cukup, baru scale out (horizontal). Prasyarat scale out: stateless services dan load balancer.

LOAD BALANCER ALGORITHMS
  • • Round-robin — default, fair distribution
  • • Least connections — untuk long-lived connections
  • • IP hash — sticky session tanpa cookie
  • • Weighted — heterogeneous instances
STATELESS PATTERN
# session di server (BAD)
app.session = { userId: 42 }

# session di client / shared store
# (GOOD — bisa scale out)
JWT cookie / Redis session store
SHARDING STRATEGIES
  • • Range-based — shard by userId 0-999, 1000-1999 (simple, hot-spot risk)
  • • Hash-based — hash(userId) % N (even distribution, sulit rebalance)
  • • Consistent hashing — minimal reshard saat tambah node (Cassandra, DynamoDB)
  • • Directory-based — lookup table (flexible, tapi single point)
KONSEP 3/3

CACHE
STRATEGIES

Cache adalah trade-off antara freshness dan latency. Pilih strategi berdasarkan tolerance stale data dan write vs read ratio.

🔄

Cache-Aside (Lazy Loading)

App baca cache dulu, miss → fetch DB → populate cache. Default untuk kebanyakan web app.

val = cache.get(key) ?? db.get(key).tap(v => cache.set(key, v, ttl))
✍️

Write-Through

Setiap write masuk cache + DB serentak. Cache selalu fresh, write latency lebih tinggi.

cache.set(key, val); db.set(key, val);  // atomic ideally
⏭️

Write-Behind (Write-Back)

Write ke cache, flush ke DB async. Throughput tinggi, risiko data loss kalau cache crash.

cache.set(key, val); queue.push(key);  // worker flush nanti
🌍

CDN / Edge Cache

Static asset dan GET response publik di-cache di edge (Cloudflare, CloudFront, Fastly).

Cache-Control: public, s-maxage=3600, stale-while-revalidate=86400
📝 EVICTION POLICIES

LRU (Least Recently Used) — default. LFU — untuk hot keys yang access pattern skewed. TTL — wajib untuk avoid stale forever.

BACK-OF-ENVELOPE

CAPACITY
ESTIMATION

Sebelum pilih database atau infra, hitung dulu order of magnitude. Jangan over-engineer untuk scale yang belum kamu butuh.

🧮 EXAMPLE: 1M DAU app
# Asumsi
DAU = 1,000,000
Avg requests/user/day = 20
Peak = 3× average

# Calculation
Total req/day = 20M
Avg QPS       = 20M / 86400 ≈ 230
Peak QPS      ≈ 700

# Storage
Avg payload = 2 KB
Daily       = 20M × 2KB = 40 GB
Yearly      = 14.6 TB
📏 LATENCY NUMBERS
L1 cache       : 1 ns
L2 cache       : 5 ns
RAM access     : 100 ns
SSD random     : 150 μs
HDD seek       : 10 ms
Same DC network: 0.5 ms
SF → NYC       : 40 ms
SG → NYC       : 220 ms

# Rule of thumb
1 ms = 1,000 μs = 1M ns
📝 POWERS OF TWO
  • • 2^10 ≈ 1 thousand (KB)
  • • 2^20 ≈ 1 million (MB)
  • • 2^30 ≈ 1 billion (GB)
  • • 2^40 ≈ 1 trillion (TB)

QUIZ • VALIDATE

TEST
PEMAHAMAN

Kamu build sistem inventory untuk e-commerce. Stock decrement harus akurat, tidak boleh oversell. Network partition terjadi antara 2 region. Apa trade-off paling tepat?

A. Pilih CP — sistem refuse write saat partisi, lebih baik downtime daripada oversell
B. Pilih AP — selalu available, rekonsiliasi stock belakangan
C. Pilih CA — tidak usah worry partisi, toh jarang terjadi
D. Gunakan Redis AP mode untuk semua state

Jawabannya A — untuk inventory, oversell = refund + marah pelanggan + kompensasi. Konsistensi strict lebih penting daripada 100% availability. Write dengan majority quorum (CP) cocok; boleh fallback read-only mode saat partisi.
