KONSEP 1/3

SOLID:
5 PRINSIP

S

Single Responsibility Principle

Satu class = satu alasan untuk berubah. Kalau UserService dipanggil untuk auth, billing, dan email, itu tiga alasan berbeda → split.

O

Open/Closed Principle

Open untuk extension, closed untuk modification. Tambah payment provider baru = tambah class, bukan edit switch.

L

Liskov Substitution Principle

Subclass harus bisa menggantikan parent tanpa surprise. Kalau Square extends Rectangle bikin setWidth() berubah sifat, itu violation.

I

Interface Segregation

Jangan paksa klien implement method yang tidak dipakai. Interface kecil lebih baik dari interface gendut.

D

Dependency Inversion

Depend pada abstraction, bukan concrete. High-level modul tidak bergantung pada low-level modul.

KONSEP 2/3

COMPOSITION
OVER INHERITANCE

Inheritance membuat tight coupling ke parent class. Perubahan di parent bisa pecah banyak subclass. Composition (has-a) lebih fleksibel dan testable.

❌ INHERITANCE TRAP
class Animal { move() {} }
class Bird extends Animal { fly() {} }
class Penguin extends Bird {
  // penguins cannot fly!
  fly() { throw new Error("LSP violation"); }
}
✅ COMPOSITION
interface Mover { move(): void; }
interface Flyer { fly(): void; }

class Bird implements Mover, Flyer { ... }
class Penguin implements Mover { ... }
// no forced inheritance of fly()
DEPENDENCY INJECTION — TESTABLE SEAM
// ❌ Hard-coded, tidak bisa di-mock
class OrderService {
  save(order) { MySQL.insert(order); EmailClient.send(...); }
}

// ✅ Injected, bisa swap untuk test
class OrderService {
  constructor(private db: Database, private mailer: Mailer) {}
  save(order) { this.db.insert(order); this.mailer.send(...); }
}

// test
const svc = new OrderService(fakeDB, fakeMailer);
KONSEP 3/3

ANTI-
PATTERNS

Code smell yang umum. Kalau kamu spot salah satu, minimal tambahkan TODO: refactor — [pattern name] dan jadwalkan cleanup.

👹

God Object

Satu class yang tahu segala hal. Ciri: file > 1000 baris, import > 30 modul, method > 40. Solusi: decompose per responsibility, extract service.

💥

Shotgun Surgery

Satu perubahan requirement → edit 15 file berbeda. Solusi: ada concept yang tersebar — konsolidasi jadi satu abstraction (sering perlu extract service atau strategy).

👀

Feature Envy

Method X di class A lebih sering panggil data dari class B daripada data class A sendiri. Solusi: pindahkan method X ke class B.

🎭

Primitive Obsession

Representasi domain pakai primitive (string email, number userId). Solusi: value object dengan validation (Email, UserId).

SIDE-BY-SIDE

OCP IN ACTION:
STRATEGY PATTERN

❌ SWITCH EXPLOSION
function charge(order, method) {
  if (method === 'stripe') {
    // 30 lines stripe logic
  } else if (method === 'paypal') {
    // 25 lines paypal logic
  } else if (method === 'midtrans') {
    // 40 lines midtrans logic
  }
  // every new provider = edit this
}
✅ STRATEGY
interface PaymentProvider {
  charge(order): Promise<Receipt>;
}

class StripeProvider implements PaymentProvider {...}
class PayPalProvider implements PaymentProvider {...}

class Checkout {
  constructor(private provider: PaymentProvider) {}
  pay(order) { return this.provider.charge(order); }
}
// tambah provider = tambah class

CLAUDE CODE

2. Prinsip yang dilanggar (SRP/OCP/LSP/ISP/DIP)

4. Concrete refactor suggestion

TEST
PEMAHAMAN

Kamu punya ReportGenerator dengan method generatePDF(), generateCSV(), dan generateExcel(). Sekarang datang requirement untuk format Parquet. Prinsip SOLID mana yang paling relevan dan apa solusi terbaik?

A. LSP — tambah method generateParquet() ke class yang sama
B. OCP — ubah jadi ReportFormatter interface, setiap format = class baru
C. DIP — inject console.log sebagai dependency
D. ISP — hapus method lain biar cuma Parquet

Jawabannya B — OCP. Tambah format harusnya open for extension (class baru) tanpa modification (edit class lama). Pattern yang cocok: Strategy — ReportFormatter interface dengan implementasi per format.
