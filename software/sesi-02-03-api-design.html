KONSEP 1/3

REST:
NOUNS + HTTP VERBS

REST resource modeling — setiap URL adalah noun, HTTP method adalah verb. Stateless, cacheable, uniform interface.

❌ RPC-STYLE (BURUK)
POST /getUser?id=42
POST /createUser
POST /updateUserName
POST /deleteUser?id=42

Verbs di URL, method selalu POST, tidak cacheable.

✅ RESTFUL
GET    /users/42
POST   /users
PATCH  /users/42
DELETE /users/42
GET    /users?status=active&page=2

Noun resources, standard methods, cacheable, predictable.

KONSEP 2/3

GRAPHQL:
CLIENT-DRIVEN QUERIES

Client minta exactly field yang dibutuhkan. Satu endpoint, schema yang strongly-typed, dan introspection. Trade-off: N+1 trap jika resolver tidak dioptimasi.

SCHEMA + QUERY
# Schema
type User { id: ID!, name: String!, posts: [Post!]! }
type Post { id: ID!, title: String!, body: String! }

# Query (client picks fields)
query {
  user(id: "42") {
    name
    posts { title }   # body tidak diambil
  }
}
⚠️ N+1 Trap: resolver untuk posts akan dipanggil sekali per user. Gunakan DataLoader untuk batch request ke database.
KONSEP 3/3

GRPC:
BINARY + STREAMING

Protocol Buffers untuk efisiensi binary, HTTP/2 untuk multiplexing, dan native streaming (unary, server-stream, client-stream, bidirectional). Cocok untuk service-to-service, tidak ideal untuk public browser API.

.proto DEFINITION
syntax = "proto3";

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
  rpc StreamUsers(Empty) returns (stream User);
}

message User {
  string id = 1;
  string name = 2;
  int64 created_at = 3;
}

KAPAN
PAKAI YANG MANA

REST

✓ Public API
✓ Caching penting
✓ Team familiar HTTP
✗ Over-fetching umum
✗ Multiple round trips

GraphQL

✓ Client diverse
✓ Nested data
✓ Rapid frontend
✗ Caching kompleks
✗ N+1 risk

gRPC

✓ Service-to-service
✓ Streaming
✓ Low-latency binary
✗ Browser support terbatas
✗ Debugging lebih sulit

DENGAN AI TOOLS

API design melibatkan banyak boilerplate (schema, client SDK, docs). AI tools memangkas waktu ini drastis.

🧠

Claude Code — Generate OpenAPI Spec

CLAUDE CODE

Dari requirement bahasa natural, Claude Code bisa generate OpenAPI 3.1 spec lengkap dengan schemas, paths, dan error responses.

# Claude Code prompt
Generate an OpenAPI 3.1 spec for a task management API:
- Resources: tasks, lists, users
- Auth: Bearer JWT
- Pagination: cursor-based
- Error shape: RFC 7807 Problem Details
Save to api/openapi.yaml and include example requests.
⚡

Cursor — Scaffold GraphQL Resolvers

CURSOR

Dari schema.graphql, Cursor Composer bisa scaffold resolvers lengkap dengan DataLoader untuk mencegah N+1.

# Cursor Composer prompt
Given schema.graphql in this repo, scaffold TypeScript resolvers
in src/resolvers/. Use DataLoader for all list relations to
prevent N+1. Add unit tests for User.posts resolver.
🤖

Copilot — Auto-Generate Client SDK

COPILOT

Dari OpenAPI spec, Copilot Chat dapat generate typed client SDK untuk bahasa tujuan.

# Copilot Chat prompt
@workspace Using api/openapi.yaml, generate a typed TypeScript
client SDK in packages/client/. Include: fetch-based http layer,
zod runtime validation, retry with backoff, and one integration test.
⚠️ Pitfall: AI-generated OpenAPI specs sering kurang pada security schemes dan error responses. Selalu review manual untuk auth, rate limit, dan edge-case status codes.

TEST
PEMAHAMAN

Tim mobile app kamu butuh API dengan nested data (user → posts → comments → authors) dan mobile punya bandwidth terbatas. Mana pilihan paling cocok?

A. REST dengan HATEOAS — client follow link
B. GraphQL — client query exact field yang dibutuhkan
C. gRPC — binary lebih kecil dari JSON
D. REST dengan multiple endpoints, client merge di mobile

Jawabannya B — GraphQL membiarkan client minta nested data dalam satu request dan hanya field yang dibutuhkan, menghemat bandwidth mobile. gRPC pilihan valid untuk service-to-service, tapi browser/mobile support dan caching HTTP jadi trade-off.
