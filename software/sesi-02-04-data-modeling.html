KONSEP 1/3

NORMALIZED
SQL SCHEMA

3NF — setiap fakta disimpan sekali. Foreign keys untuk relasi, indexes untuk query path yang sering. Denormalize hanya saat profiling nunjukin problem.

CREATE TABLE users (
  id           UUID PRIMARY KEY,
  email        TEXT UNIQUE NOT NULL,
  created_at   TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE orders (
  id           UUID PRIMARY KEY,
  user_id      UUID REFERENCES users(id),
  total_cents  INTEGER NOT NULL,
  status       TEXT NOT NULL
);

CREATE INDEX ix_orders_user_status ON orders(user_id, status);
KONSEP 2/3

DOCUMENT STORE:
ACCESS PATTERN FIRST

Document DB (MongoDB, DynamoDB) memilih schema berdasarkan how you read, bukan how you write. Denormalisasi OK; duplication untuk locality.

// MongoDB — embed comments in posts (1:N, bounded)
{
  _id: "post_42",
  title: "Clean Code 101",
  author: { id: "u_7", name: "Alice" },  // denormalized
  comments: [
    { id: "c_1", body: "Great!", author: { name: "Bob" } }
  ]
}
⚠️ Denormalization trade-off: update author name butuh fanout ke semua dokumen. OK jika read >>> write.
KONSEP 3/3

EVENT SOURCING
+ CQRS

Simpan events, bukan state. Current state = projection dari event log. Kuat untuk audit trail, temporal queries, dan CQRS read models.

// Event log (append-only)
{ ts: "2026-04-15T10:00", type: "AccountCreated", id: "a_1", owner: "u_7" }
{ ts: "2026-04-15T10:05", type: "Deposited", id: "a_1", amount: 100 }
{ ts: "2026-04-15T10:10", type: "Withdrawn", id: "a_1", amount: 30 }

// Current state (projection)
{ id: "a_1", owner: "u_7", balance: 70 }

Copilot — Query + Index Hints

COPILOT

QUIZ

TEST
PEMAHAMAN

Kamu build banking ledger yang butuh full audit trail, temporal queries ("berapa saldo di 15 Januari?"), dan compliance. Model data paling cocok?

A. Normalized SQL dengan audit_log table terpisah
B. Event sourcing — transactions sebagai append-only events
C. Document DB dengan nested transaction array
D. Key-value store dengan balance snapshots

Jawaban B — event sourcing cocok untuk banking ledger karena transactions inherently append-only, audit trail built-in, dan replay event log memberi temporal query natural. SQL + audit_log bisa jalan tapi duplikasi effort dan risk drift antara source-of-truth dan audit.
