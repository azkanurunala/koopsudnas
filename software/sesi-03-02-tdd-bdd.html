KONSEP 1/3

RED →
GREEN → REFACTOR

Disiplin TDD: tidak boleh menulis production code tanpa failing test dulu. Siklus idealnya pendek — 2-5 menit per putaran.

1. RED
test('fizz for 3', () => {
  expect(fizzbuzz(3))
    .toBe('Fizz');
});
// ❌ ReferenceError

Tulis test yang gagal. Fungsi belum ada.

2. GREEN
function fizzbuzz(n) {
  return 'Fizz';
}
// ✅ passes (cheating!)

Kode minimum untuk hijau — boleh "curang" dulu.

3. REFACTOR
function fizzbuzz(n) {
  if (n % 3 === 0) return 'Fizz';
  return String(n);
}
// ✅ general solution

Setelah hijau, generalisasi. Test tetap hijau = safe.

📝 THREE LAWS OF TDD (Uncle Bob)
  • • Jangan tulis production code kecuali untuk membuat failing test pass
  • • Jangan tulis lebih banyak test daripada yang diperlukan untuk failing
  • • Jangan tulis lebih banyak production code daripada yang diperlukan untuk pass
KONSEP 2/3

BDD:
GIVEN-WHEN-THEN

BDD (Behavior-Driven Development) menggeser bahasa test ke ubiquitous language domain — stakeholder bisa baca dan validasi behavior, bukan cuma engineer.

❌ TECH-CENTRIC TEST
test('handler returns 402', async () => {
  const res = await POST('/checkout', { cart: [], user: 'u1' });
  expect(res.status).toBe(402);
});
✅ BEHAVIOR-CENTRIC (Gherkin / Vitest describe)
describe('Checkout', () => {
  describe('given user with empty cart', () => {
    it('when user proceeds to pay, then payment is rejected', async () => {
      const user = aUser().withEmptyCart().build();
      const result = await checkout(user);
      expect(result.rejected).toBe(true);
      expect(result.reason).toBe('empty_cart');
    });
  });
});

Business analyst bisa baca, reviewer QA bisa validasi intent.

KONSEP 3/3

PROPERTY-BASED
TESTING

Example-based test menguji contoh konkret. Property-based test menguji invariant yang berlaku untuk semua input — framework generate ratusan input random dan shrinks ke minimal failing case.

EXAMPLE: SORT FUNCTION INVARIANTS
import fc from 'fast-check';

test('sort is idempotent', () => {
  fc.assert(
    fc.property(fc.array(fc.integer()), (arr) => {
      const once = sort(arr);
      const twice = sort(once);
      expect(twice).toEqual(once);
    })
  );
});

test('sort preserves length', () => {
  fc.assert(fc.property(fc.array(fc.integer()),
    (arr) => sort(arr).length === arr.length));
});

Common Invariants

  • • Idempotent: f(f(x)) = f(x)
  • • Inverse: decode(encode(x)) = x
  • • Commutative: f(a,b) = f(b,a)
  • • Model-based: matches reference impl

Tools

  • • JS/TS: fast-check
  • • Python: hypothesis
  • • Rust: proptest, quickcheck
  • • Java: jqwik
SIDE-BY-SIDE

OUTSIDE-IN VS
INSIDE-OUT

🌐 OUTSIDE-IN (London School)

Mulai dari acceptance test di level UI/API. Turun per lapisan, mock collaborator yang belum ada.

// 1. E2E: "user can checkout"
// 2. Controller test (mock service)
// 3. Service test (mock repo)
// 4. Repo test (real DB)
  • ✓ Drive dari user value
  • ✓ Banyak mock di awal
  • ✗ Risiko over-mocking
🔬 INSIDE-OUT (Chicago School)

Mulai dari domain object terdalam. Real object semua. Integrasi bottom-up.

// 1. Domain entity (pure)
// 2. Service (real entity)
// 3. Controller (real service)
// 4. E2E last
  • ✓ Zero mock di core
  • ✓ Design muncul organik
  • ✗ Bisa miss user intent
AI-ASSISTED WORKFLOW

TDD
DENGAN AI TOOLS

AI tools menambah kecepatan siklus red-green-refactor. Tapi hati-hati: jangan biarkan AI skip fase red. Kamu yang tulis test, AI yang bantu implementasi.

🧠

Claude Code — Spec → Failing Test

CLAUDE CODE

Paste user story atau acceptance criteria dari Jira, minta Claude Code tulis failing test dulu:

# Claude Code prompt
Acceptance criteria:
"Given user with 3 items in cart totalling Rp 200.000, when user
applies coupon WELCOME10, then total becomes Rp 180.000."

Write a failing Vitest test in tests/checkout.test.ts. Use
Given-When-Then describe/it structure. Do NOT implement the
function yet — I want to see red first.
⚡

Cursor Composer — Red → Green Sweep

CURSOR

Setelah failing test jadi, panggil Cursor Composer untuk tulis implementasi minimum:

# Cursor Composer prompt
The tests in tests/checkout.test.ts are red. Implement
applyCoupon() in src/checkout.ts with the MINIMUM code needed
to make all tests pass. Don't add extra features. After, run
`npm test` and confirm green. Then suggest one refactor.
🤖

Copilot — Edge Case Generation

COPILOT

Setelah 2-3 test happy path, minta Copilot Chat identify edge case yang kamu miss:

# Copilot Chat prompt
/tests Review the existing tests for applyCoupon() and suggest
5 edge cases I might have missed: expired coupon, coupon code
casing, zero-total cart, coupon stacking, and currency rounding.
Write each as a new describe block following Given-When-Then.
⚠️ Pitfall: AI gampang tergoda tulis implementasi bareng test. Pegang disiplin: "red first, then green". Kalau test hijau dari awal, test itu tidak bernilai.
QUIZ • VALIDATE

TEST
PEMAHAMAN

Dalam siklus TDD, setelah failing test kamu jadi hijau dengan kode minimum ("cheating" return hardcoded), apa yang harus kamu lakukan berikutnya?

A. Commit dan lanjut ke fitur berikutnya — test sudah hijau
B. Tambah test baru yang memaksa generalisasi, atau refactor implementasi agar benar-benar correct
C. Delete test dan tulis ulang dengan implementasi yang benar sekaligus
D. Skip refactor phase — refactor hanya dilakukan di akhir sprint

Jawabannya B — cheating adalah langkah sah di TDD tapi hanya sementara. Kamu punya dua jalur: tambah test yang bikin cheating pecah (forces generalization), atau refactor implementasi sekarang. Jangan pernah commit cheating tanpa salah satunya.

📝 Task: String Calculator Kata

Bangun fungsi add(numbers: string): number lewat TDD murni. Tulis SATU test red per langkah, hijau-kan, refactor, baru lanjut.

SPEC (tambah satu per satu)
  1. add("") → 0
  2. add("1") → 1
  3. add("1,2") → 3
  4. add("1,2,3,4,5") → 15
  5. add("1\n2,3") → 6 (newline sebagai delimiter)
  6. add("//;\n1;2") → 3 (custom delimiter)
  7. Negatif → throw Error dengan daftar negative numbers
🤖 AI PROMPT TEMPLATE
Guide me through the String Calculator kata using strict TDD.
For each step, write ONE failing Vitest test. Wait for me to
confirm red, then write the minimum code to make it green.
Then refactor if needed. Then move to the next spec item.
Never skip ahead. Start with step 1: add("") → 0.
