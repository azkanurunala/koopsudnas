KONSEP 1/3

TONE:
CODE, BUKAN CODER

Review yang baik membuat author mau perbaiki, bukan defensif. Gunakan konvensi label: blocking (harus di-fix), suggestion (pertimbangan), nit (opini minor).

❌ TONE BURUK
"Kenapa pake var di sini?
Jelek banget. Tolong benerin."

"Kode kamu bakal error kalo
null. Masa gini aja kelewat."

"Function ini kepanjangan,
gak efisien."

Menyerang orang, tidak actionable, tidak jelas level urgensi.

✅ TONE BAIK
[blocking] Kalau `user` null di
baris 42, line 45 akan throw.
Bisa tambah guard clause?

[suggestion] Bisa di-extract ke
`calcTotal()` — reusable di
OrderService juga.

[nit] Konsisten: kita pakai
`const` di file lain.

Fokus code, actionable, label jelas sehingga author bisa prioritize.

📝 RULES OF ENGAGEMENT
  • • Selalu asumsikan author punya konteks yang kamu tidak tahu
  • • Tanya dulu sebelum claim ("Apa pertimbangan pakai X di sini?")
  • • Approve kalau hanya ada nit — biar author yang memutuskan merge
  • • Diskusi panjang (>3 round) → pindah ke sync call / huddle
KONSEP 2/3

LINTER:
AUTOMATE THE BORING

Setiap debate style di PR adalah waste. Setup linter + formatter sekali, lupakan selamanya. Reviewer bisa fokus ke logic dan design, bukan spasi vs tab.

🟨

JavaScript/TypeScript

  • • ESLint + typescript-eslint
  • • Biome — fast all-in-one (Rust)
  • • Prettier — format only
🐍

Python

  • • Ruff — 100x faster than pylint
  • • Pylint — comprehensive but slow
  • • Black — opinionated formatter
🦀

Rust

  • • Clippy — built-in, very strict
  • • rustfmt — canonical formatter
🎯

Type Checkers

  • • tsc --strict
  • • mypy --strict / pyright
  • • flow (Meta)
EXAMPLE: BIOME CONFIG
// biome.json
{
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true,
      "correctness": { "noUnusedVariables": "error" },
      "suspicious": { "noExplicitAny": "error" }
    }
  },
  "formatter": { "indentStyle": "space", "indentWidth": 2 }
}
KONSEP 3/3

PRE-COMMIT
SAFETY NET

Pre-commit hooks jalan sebelum commit masuk repo. Dengan husky + lint-staged, kamu run linter + formatter + type-check hanya di file yang di-stage, cepat dan fokus.

SETUP: package.json
{
  "scripts": {
    "prepare": "husky install"
  },
  "lint-staged": {
    "*.{ts,tsx}": [
      "biome check --write",
      "tsc --noEmit"
    ],
    "*.{md,json}": ["biome format --write"]
  }
}

# .husky/pre-commit
npx lint-staged

✅ Yang layak di pre-commit

  • • Linter auto-fix
  • • Formatter
  • • Fast type-check
  • • Leaked secret scanner (gitleaks)

❌ Jangan di pre-commit

  • • Full test suite (terlalu lambat)
  • • E2E / Playwright
  • • Docker build
  • • Deploy staging
SIDE-BY-SIDE

MANUAL REVIEW VS
AUTOMATED PIPELINE

❌ TANPA AUTOMATION
  • • Reviewer debat spasi/tab di PR
  • • Typo unused variable lolos ke main
  • • any di TypeScript tidak ter-detect
  • • Credentials ter-commit ke git history
  • • Style inconsistent antar file
  • • PR 400 komentar, 350 di antaranya nit

Reviewer fatigue, author frustrated, merge delayed.

✅ DENGAN AUTOMATION
  • • Pre-commit: biome + tsc + gitleaks
  • • CI: lint + typecheck + test + coverage
  • • Danger.js: auto-comment di PR
  • • CodeRabbit / Copilot Review: first pass
  • • Reviewer cuma fokus design & logic
  • • Style = konsisten by construction

PR kecil, reviewer fresh, time-to-merge < 4 jam.

AI-ASSISTED WORKFLOW

CODE REVIEW
DENGAN AI TOOLS

AI reviewer tidak menggantikan manusia, tapi first pass yang bagus. Kamu fokus ke hal yang AI belum bisa: intent, trade-off, strategi product.

🐇

CodeRabbit / Copilot Review — First-Pass

CODERABBIT

Integrate CodeRabbit sebagai GitHub App. Untuk Copilot Review, enable di repo settings. Di PR baru, auto-scan jalan dan comment:

# .coderabbit.yaml
reviews:
  profile: chill
  request_changes_workflow: false
  high_level_summary: true
  poem: false
  review_status: true
  auto_review:
    enabled: true
    drafts: false
language: id
🧠

Claude Code — /review-pr Skill

CLAUDE CODE

Claude Code punya built-in review-pr skill. Dari terminal lokal, jalankan:

# Claude Code prompt
/review-pr 1247

# Or manual with specific focus
Fetch PR #1247 from origin. Review with focus on:
1. Security (SQL injection, XSS, secrets)
2. Performance (N+1 queries, loops in hot path)
3. Error handling (unhandled rejections, silent catch)
Label each finding as [blocking] / [suggestion] / [nit].
⚡

Cursor — Bulk Lint Auto-Fix

CURSOR

Ketika kamu enable strict rule baru dan dapat 200+ lint error, minta Cursor Composer fix bulk:

# Cursor Composer prompt
Run `biome check --write src/` and fix all auto-fixable errors.
For the remaining errors that need manual judgment (e.g.,
noExplicitAny), list them by file:line with suggested fix.
Don't touch tests/ for now. Commit as "chore: bulk lint fix".
⚠️ Pitfall: AI reviewer kadang complain tentang hal yang sebenarnya intentional (misal legacy compat). Author boleh reply "wontfix — intentional because X" dan AI harus dismiss.
QUIZ • VALIDATE

TEST
PEMAHAMAN

Di PR kamu, reviewer meninggalkan komentar: "Kode ini jelek, kenapa gak dipikir dulu sebelum submit?" Apa respons paling profesional?

A. Balas dengan tone sama: "Kamu juga gak ngerti konteks, baca dulu"
B. Close PR, rewrite dari awal sendirian untuk menghindari konflik
C. Tanya spesifik apa yang harus diperbaiki, escalate ke tech lead soal tone komentar
D. Ignore komentar, merge langsung karena CI sudah hijau

Jawabannya C — minta klarifikasi actionable ("spesifiknya bagian mana?") sambil tandai bahwa tone komentar tidak sehat ke lead. Eskalasi ke manusia, bukan perang di thread PR. Kode review boleh kritis, tapi tidak boleh merendahkan.

LATIHAN
SETUP PIPELINE

📝 Task

Setup full quality pipeline di proyek kamu: Biome untuk lint+format, TypeScript strict, husky + lint-staged pre-commit, dan gitleaks untuk secret scanning.

  1. npm i -D @biomejs/biome husky lint-staged
  2. Tambahkan "strict": true di tsconfig
  3. Jalankan npx husky init dan edit .husky/pre-commit
  4. Konfigurasi lint-staged di package.json
  5. Install gitleaks binary dan jalankan gitleaks protect --staged
  6. Buat 1 file dengan error sengaja → commit → verify pre-commit block
