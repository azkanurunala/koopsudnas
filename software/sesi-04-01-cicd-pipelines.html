KONSEP 1/3

PIPELINE
STAGES

Setiap pipeline modern punya sekuens yang bisa diprediksi: build → test → scan → package → deploy. Setiap stage adalah gate — gagal di satu stage berarti tidak lanjut.

GITHUB ACTIONS — MINIMAL VIABLE PIPELINE
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - run: npm test -- --coverage
      - run: npm run lint
      - uses: github/codeql-action/analyze@v3
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

📦 Stage Order Matters

  • 1. Lint/format (cepat, fail early)
  • 2. Build (compile, typecheck)
  • 3. Test (unit, integration)
  • 4. Security scan (SAST, deps)
  • 5. Package (Docker, artifact)
  • 6. Deploy (staging → prod)

⏱️ Target Budgets

  • • Lint + typecheck: <2 min
  • • Unit test: <5 min
  • • Full PR pipeline: <10 min
  • • Deploy to staging: <15 min
  • • Lebih lambat = merge anxiety
KONSEP 2/3

MATRIX
PARALLELIZATION

Kalau kamu test 3 Node versions × 3 OS, sequential makan 9× waktu. Matrix build fork jadi 9 runner paralel — 9× speedup tanpa nambah effort.

MATRIX EXAMPLE — CROSS-PLATFORM LIBRARY
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node: [18, 20, 22]
        include:
          - os: ubuntu-latest
            node: 22
            experimental: true
        exclude:
          - os: windows-latest
            node: 18
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test

8 jobs (3×3 - 1 excluded) jalan paralel. fail-fast: false supaya lihat semua kegagalan sekaligus.

KONSEP 3/3

SECRETS
& OIDC

Long-lived AWS/GCP keys di GitHub Secrets = risk. OIDC federation: GitHub issue short-lived token, AWS trust GitHub, zero stored credentials.

❌ LONG-LIVED ACCESS KEY
env:
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_KEY }}
  AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET }}

# Bocor = full account access
# Rotasi manual, sering lupa
✅ OIDC FEDERATION
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123:role/gh-deploy
          aws-region: ap-southeast-1
      - run: aws s3 sync ./dist s3://my-bucket

Token valid ~1 jam, scoped ke repo ini saja. Bocor = minimal blast radius.

  image: node:20
  script:

  • ✓ Include/extends reusable
  • ✓ Rules lebih ekspresif
  • ✗ Marketplace tidak seluas

QUIZ • VALIDATE

TEST
PEMAHAMAN

Tim kamu store AWS access key di GitHub Secrets dan pakai untuk deploy. Key bocor lewat PR yang tidak sengaja print env. Apa solusi jangka panjang paling tepat?

A. Rotate key manual setiap minggu, tambah code review lebih ketat
B. Migrasi ke OIDC federation — GitHub issue short-lived token, zero stored credentials
C. Encrypt secret dengan GPG sebelum commit ke repo
D. Pindahkan deploy ke manual — hanya tech lead yang jalankan lokal

Jawabannya B — OIDC menghilangkan akar masalah. Rotasi manual akan terlupakan, GPG di repo tetap bisa di-decrypt, dan manual deploy tidak scale. OIDC token hanya valid 1 jam dan scoped ke workflow tertentu.
