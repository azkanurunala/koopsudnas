KONSEP 1/3

DOCKERFILE:
MULTI-STAGE BUILD

Multi-stage build memisahkan build environment (berat, banyak toolchain) dari runtime image (slim, production). Hasil: image kecil, secure.

# Stage 1: build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: runtime
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node
EXPOSE 3000
HEALTHCHECK CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/server.js"]
KONSEP 2/3

K8S:
POD → DEPLOYMENT → SERVICE

Dari satu container sampai scalable service: Pod (unit terkecil) → Deployment (replica management) → Service (stable network endpoint) → Ingress (external access).

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
spec:
  replicas: 3
  selector:
    matchLabels: { app: api }
  template:
    metadata:
      labels: { app: api }
    spec:
      containers:
        - name: api
          image: ghcr.io/org/api:v1.2.3
          ports: [{ containerPort: 3000 }]
          livenessProbe: { httpGet: { path: /health, port: 3000 } }
          readinessProbe: { httpGet: { path: /ready, port: 3000 } }
KONSEP 3/3

HELM:
TEMPLATING + VALUES

Helm chart memisahkan template (struktur) dari values (config per environment). Satu chart untuk dev, staging, prod.

# values-prod.yaml
replicas: 10
image:
  tag: v1.2.3
resources:
  requests: { cpu: 500m, memory: 512Mi }
  limits:   { cpu: 1000m, memory: 1Gi }

# templates/deployment.yaml
replicas: {{ .Values.replicas }}
image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
SIDE-BY-SIDE

BLOATED VS
SLIM IMAGE

❌ BAD (1.2GB)
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y \
    nodejs npm git vim curl build-essential
WORKDIR /app
COPY . .
RUN npm install
CMD ["npm", "start"]

• Full OS base
• Build tools in runtime
• Runs as root
• No health check

✅ GOOD (120MB)
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
USER node
HEALTHCHECK CMD wget -qO- :3000/health
CMD ["node", "dist/server.js"]

• Alpine base
• Multi-stage (no build tools)
• Non-root user
• Health check

PEMAHAMAN

Pod kamu sering di-kill oleh K8s dengan OOMKilled walaupun aplikasi healthy. Apa yang paling mungkin jadi root cause?

A. Liveness probe terlalu agresif
B. Memory limit di resources.limits terlalu rendah untuk working set aplikasi
C. Service tidak punya session affinity
D. ConfigMap terlalu besar

Jawaban B — OOMKilled secara definisi berarti container melebihi memory limit. Naikkan resources.limits.memory atau kurangi memory usage aplikasi. Cek dengan kubectl top pod dan profiling.
