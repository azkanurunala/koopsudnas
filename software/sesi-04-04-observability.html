KONSEP 1/3

3 PILLARS:
LOGS, METRICS, TRACES

Tiga sumber informasi yang saling melengkapi: logs (events), metrics (measurements over time), traces (request journey across services).

📜 Logs

Discrete events with context.

{
  "ts": "2026-04-15T...",
  "level": "error",
  "trace_id": "abc123",
  "msg": "db timeout",
  "user_id": "u_7"
}

📊 Metrics

Aggregated measurements over time.

http_requests_total{
  path="/api/users",
  method="GET",
  status="200"
} 45321

🔗 Traces

Request across service boundaries.

trace_id=abc123
├─ api (50ms)
│  └─ db.query (45ms) ⚠
│     └─ db.lock_wait (40ms)
└─ cache.get (2ms)
KONSEP 2/3

OPENTELEMETRY:
UNIFIED INSTRUMENTATION

OpenTelemetry (OTel) adalah standar untuk instrument logs + metrics + traces. Satu SDK, satu API, vendor-agnostic backend.

// Node.js — automatic instrumentation
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations }
  from '@opentelemetry/auto-instrumentations-node';

const sdk = new NodeSDK({
  serviceName: 'api-service',
  instrumentations: [getNodeAutoInstrumentations()]
});
sdk.start();

// Manual span for business logic
const tracer = trace.getTracer('api');
await tracer.startActiveSpan('processPayment', async span => {
  span.setAttribute('user.id', userId);
  // ... work ...
  span.end();
});
KONSEP 3/3

SLO +
ERROR BUDGET

SLO (Service Level Objective) = target reliability yang kamu janjikan. Error budget = berapa banyak "fail" yang acceptable dalam window waktu. Alerting yang berguna = alert when burning budget too fast.

# SLO definition
service: api
sli: success_rate (ratio of 2xx+3xx / total requests)
slo: 99.9% over 30-day rolling window
error_budget: 0.1% = 43 minutes of downtime per 30 days

# Burn rate alert (multi-window)
alert: ErrorBudgetBurnFast
expr: |
  (1 - success_rate_1h) > (14.4 * 0.001)
  AND (1 - success_rate_5m) > (14.4 * 0.001)
for: 2m
labels: { severity: page }
annotations:
  summary: Burning 2% of 30-day budget in 1h
ALERTING RULES

ACTIONABLE VS
NOISY ALERTS

❌ NOISY
  • • CPU > 80% for 5 min
  • • Memory > 70%
  • • Any 5xx in last 5 min
  • • Deployment happened

Causes alert fatigue — most are not actionable.

✅ ACTIONABLE
  • • Error budget burn rate > 14.4x (2h window)
  • • p95 latency > SLO for 10 min
  • • Saturation: queue depth growing unbounded
  • • Dependency down (DB unreachable)

Every alert requires human action. Has runbook.

🤖

Copilot — Sane Alert Rules

COPILOT

Multi-window burn rate alerts butuh math — Copilot Chat dapat generate YAML rules yang tepat.

# Copilot Chat prompt
Generate Prometheus alert rules for SLO 99.9% availability on api:
- Multi-window burn rate (fast: 1h at 14.4x, slow: 6h at 6x)
- Only page on fast burn
- Include sane `for:` durations to avoid flapping
- Add runbook_url annotation pointing to
  https://runbooks/api/slo-burn
- Severity label: page for fast, ticket for slow
⚠️ Pitfall: AI-generated alerts kadang tidak punya runbook. Alert tanpa runbook = panic saat pager berbunyi. Selalu tambahkan runbook_url annotation.
QUIZ

TEST
PEMAHAMAN

Pager kamu dipanggil 20x semalam karena alert "CPU > 80%". Aplikasi tetap healthy, user tidak complain. Apa yang harus kamu lakukan?

A. Scale up cluster untuk menurunkan CPU baseline
B. Delete alert — CPU bukan SLI yang user-facing; ganti dengan error budget burn rate
C. Naikkan threshold ke 90%
D. Silence alert semalaman

Jawaban B — CPU adalah resource metric, bukan SLI. Alert harus pada user-visible symptoms. Ganti ke error budget burn rate atau p95 latency. Menaikkan threshold atau silence = menutupi masalah yang lebih fundamental (alert philosophy yang salah).
