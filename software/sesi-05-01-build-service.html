📝 DESIGN DOC TEMPLATE
# Design Doc: URL Shortener Service

## Problem
Tim internal butuh short link dengan analytics dasar (click count, referrer).

## Goals
- Shorten URL < 100ms p95
- Redirect latency < 50ms p95
- Track click events async

## Non-Goals
- Custom domain per user
- A/B testing
- Real-time dashboard

## API Surface
POST /api/links        => { slug, originalUrl }
GET  /:slug            => 302 redirect
GET  /api/links/:slug  => analytics summary

## Data Model
links(slug PK, original_url, created_at, created_by)
click_events(id, slug FK, ts, referrer, user_agent)

## Risks
- Slug collision (mitigate: base62 + retry)
- Hot redirect path (mitigate: in-memory cache)
KONSEP 2/3

ARCHITECTURE
DECISION RECORDS

Setiap keputusan arsitektur yang susah di-revert layak dicatat sebagai ADR. Format-nya pendek, tapi disiplin — context, decision, consequences. Disimpan di docs/adr/ berdampingan dengan code.

📄 ADR-0001: Pilih PostgreSQL untuk primary datastore
# ADR-0001: Use PostgreSQL as Primary Datastore

## Status
Accepted — 2026-04-15

## Context
Kita butuh durable storage untuk links + click_events. Click volume
diestimasi 10k/hari di tahun pertama. Transactional consistency penting
untuk slug allocation (no duplicates).

## Decision
Pakai PostgreSQL 16 via Supabase managed service.

## Alternatives Considered
- DynamoDB — NoSQL, scalable, tapi tim belum familiar & query analytics ribet
- SQLite — terlalu kecil untuk concurrent writes
- MongoDB — tidak butuh fleksibilitas document

## Consequences
+ Transactional slug allocation trivial
+ Analytics query pakai SQL standard
- Vendor lock-in ringan ke Supabase
- Perlu koneksi pool management di serverless
📌 ADR RULES OF THUMB
  • • Tulis ADR hanya untuk keputusan yang susah/mahal di-revert
  • • Status: Proposed → Accepted → Superseded by ADR-XXXX
  • • Jangan edit ADR lama — tulis ADR baru yang superseded
  • • Simpan di repo sebagai docs/adr/NNNN-title.md
KONSEP 3/3

IMPLEMENTASI
DENGAN SOLID

Setelah design doc & ADR solid, implementasi jadi mechanical. Pakai layered structure sederhana — route → service → repository — dan pisahkan I/O dari business logic supaya testable.

✅ CLEAN LAYERED IMPLEMENTATION
// src/services/link-service.ts — business logic only, no I/O
export class LinkService {
  constructor(private repo: LinkRepository, private slugGen: SlugGenerator) {}

  async shorten(originalUrl: string, createdBy: string): Promise<Link> {
    validateUrl(originalUrl);
    const slug = await this.slugGen.nextAvailable();
    return this.repo.insert({ slug, originalUrl, createdBy });
  }

  async resolve(slug: string): Promise<string | null> {
    const link = await this.repo.findBySlug(slug);
    return link?.originalUrl ?? null;
  }
}

// src/routes/links.ts — HTTP concerns only
router.post('/api/links', async (req, res) => {
  const link = await linkService.shorten(req.body.url, req.user.id);
  res.status(201).json(link);
});
S — Single Responsibility

LinkService hanya urus business rules, bukan HTTP, bukan DB syntax.

D — Dependency Inversion

Service depend pada interface LinkRepository, bukan Postgres driver.

PEMAHAMAN

Kamu sudah punya design doc & ADR. Saat implementasi, kamu menyadari ADR-0001 (pilihan Postgres) ternyata kurang tepat — harusnya DynamoDB. Apa langkah yang benar?

A. Edit ADR-0001 langsung, ubah decision jadi DynamoDB
B. Hapus ADR-0001 karena sudah tidak akurat
C. Tulis ADR-0003 baru dengan status "Supersedes ADR-0001", update ADR-0001 status jadi "Superseded by ADR-0003"
D. Biarkan ADR lama, cukup ubah kode — ADR hanya formalitas

Jawabannya C — ADR adalah historical record, tidak boleh di-edit destructively. Menulis ADR baru yang "supersedes" menjaga history keputusan dan context kenapa kamu pernah berpikir lain, yang sangat berharga saat onboarding anggota tim baru atau saat debug keputusan di masa depan.
