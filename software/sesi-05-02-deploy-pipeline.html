KONSEP 1/3

MULTI-STAGE
DOCKERFILE

Dockerfile yang baik punya tiga properti: kecil (final image < 200MB), cepat (leverage layer caching), dan aman (non-root, minimal dependencies). Pakai multi-stage build.

❌ NAIVE DOCKERFILE
FROM node:20
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
  • • 1.2 GB image (base image besar)
  • • Include devDependencies di production
  • • Run as root
  • • Cache miss setiap source change
✅ MULTI-STAGE
# Stage 1: build
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build && npm prune --production

# Stage 2: runtime
FROM node:20-alpine
RUN adduser -D app
USER app
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/index.js"]
📌 LAYER CACHING TRICK

Copy package*.json sebelum copy source. Saat source berubah tapi dependencies tidak, npm ci layer akan hit cache dan build jadi ~10x lebih cepat.

KONSEP 2/3

CI/CD
PIPELINE DESIGN

Pipeline yang baik fail fast — lint sebelum test, test sebelum build, build sebelum deploy. Setiap stage cache output-nya, dan deploy hanya jalan di branch main.

⚙️ .github/workflows/ci-cd.yml
name: CI/CD
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - run: npm run lint
      - run: npm test -- --coverage

  build-push:
    needs: lint-test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write, id-token: write }
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with: { registry: ghcr.io, username: ${{ github.actor }}, password: ${{ secrets.GITHUB_TOKEN }} }
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build-push
    runs-on: ubuntu-latest
    environment: production
    steps:
      - run: gcloud run deploy url-shortener --image ghcr.io/${{ github.repository }}:${{ github.sha }} --region asia-southeast2
KONSEP 3/3

TERRAFORM
INFRA AS MODULE

Semua infrastructure didefinisikan sebagai Terraform module reusable. Tujuannya: setup environment baru (staging, prod) hanya dengan ubah satu variable. Tidak ada click-ops di console cloud.

🏗️ modules/service/main.tf
variable "environment" { type = string }
variable "image"       { type = string }
variable "db_tier"     { type = string; default = "db-f1-micro" }

resource "google_cloud_run_v2_service" "app" {
  name     = "url-shortener-${var.environment}"
  location = "asia-southeast2"

  template {
    containers {
      image = var.image
      env {
        name  = "DATABASE_URL"
        value_source {
          secret_key_ref {
            secret  = google_secret_manager_secret.db_url.secret_id
            version = "latest"
          }
        }
      }
    }
  }
}

resource "google_sql_database_instance" "db" {
  name             = "shortener-db-${var.environment}"
  database_version = "POSTGRES_16"
  region           = "asia-southeast2"
  settings { tier = var.db_tier }
}

output "service_url" {
  value = google_cloud_run_v2_service.app.uri
}
🔄 REUSE via environments/
module "prod" {
  source      = "../modules/service"
  environment = "prod"
  image       = var.image_tag
  db_tier     = "db-custom-2-4096"
}

QUIZ • VALIDATE

TEST
PEMAHAMAN

Build Dockerfile kamu rebuild semua layer (> 3 menit) setiap kali ada perubahan kecil di kode. Apa penyebab paling mungkin dan fix-nya?

A. GitHub Actions runner terlalu lambat — pindah ke self-hosted runner
B. Dockerfile melakukan COPY . . sebelum npm ci, jadi cache layer di-invalidate tiap file change. Fix: COPY package*.json dulu, baru npm ci, baru COPY source.
C. node_modules terlalu besar — hapus saja dari dependencies
D. Pakai base image node:20 alih-alih node:20-alpine

Jawabannya B — Docker layer caching bekerja line-by-line. Kalau COPY . . ada di atas npm ci, setiap perubahan source akan invalidate cache untuk npm ci juga. Solusinya: copy package*.json duluan, run npm ci, baru copy source. Dengan ini, npm ci hit cache selama package-lock.json tidak berubah.
