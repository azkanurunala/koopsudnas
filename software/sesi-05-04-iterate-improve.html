KONSEP 1/3

BLAMELESS
POSTMORTEM

Tujuan postmortem bukan mencari siapa yang salah — tapi menjawab "sistem seperti apa yang memungkinkan error ini terjadi, dan bagaimana kita ubah sistemnya". Kalau postmortem bikin orang takut, orang akan sembunyikan insiden berikutnya.

📄 POSTMORTEM TEMPLATE
# Postmortem: 2026-04-12 — DB Connection Pool Exhaustion

## Status
Resolved

## Summary (1 paragraph)
Antara 14:03-14:47 WIB, service url-shortener mengalami 38% error rate
akibat connection pool DB exhausted. Root cause: endpoint baru
/api/links/bulk menggunakan transaksi long-running tanpa release.
Mitigasi: rollback revisi 00042. Permanent fix: connection leak patch
di PR #78.

## Impact
- Duration: 44 minutes
- User impact: ~2,100 failed requests, 3 users reported via support
- Error budget consumed: 18% of monthly availability budget
- Revenue impact: none (internal tool)

## Timeline (WIB)
14:03  Deploy revisi 00042 (bulk endpoint)
14:08  First 500 errors in logs
14:12  Alert HighErrorBudgetBurn fires
14:14  On-call (Aldi) acknowledges
14:18  Dashboard confirms db_pool_waiting spike
14:23  Decision: rollback to 00041
14:25  Rollback command executed
14:31  Error rate drops <1%
14:47  Alert clears, incident marked resolved

## Root Cause (5 Whys)
1. Mengapa error? Connection pool habis.
2. Mengapa habis? Connection di-acquire tapi tidak di-release.
3. Mengapa tidak di-release? Transaksi long-running di bulk endpoint.
4. Mengapa tidak ketahuan? Tidak ada unit test untuk resource cleanup.
5. Mengapa tidak ada test? Checklist review tidak include resource lifecycle.

## What Went Well
- Alert fire dalam 9 menit setelah symptoms
- Runbook db-pool-exhaustion.md memudahkan triage
- Rollback command work di first try

## What Went Wrong
- Load test di staging tidak exercise bulk endpoint
- Code review tidak catch transaksi tanpa cleanup

## Action Items
| # | Action                                          | Owner | Due        |
|---|-------------------------------------------------|-------|------------|
| 1 | Fix connection leak (PR #78)                    | Aldi  | 2026-04-13 |
| 2 | Tambah load test untuk bulk endpoints           | Dinda | 2026-04-18 |
| 3 | Update code review checklist (resource lifecycle)| Bayu  | 2026-04-20 |
| 4 | Add metric db_connection_held_duration          | Aldi  | 2026-04-22 |
💡 BLAMELESS LANGUAGE

Ganti "X lupa release connection" → "sistem tidak memaksa release connection". Fokus pada sistem, bukan manusia.

KONSEP 2/3

FEEDBACK
LOOP PROCESS

Feedback loop adalah saluran terstruktur yang mengalirkan sinyal dari reality kembali ke backlog. Ada 4 sumber utama yang harus kamu buka: user, ops, team, code.

👥

User Feedback

Support tickets, NPS survey, interview, usage analytics.

→ Cadence: weekly triage
📡

Ops Data

SLO burn rate, error trends, latency degradation, toil hours.

→ Cadence: daily standup + monthly review
🗣️

Team Retro

What went well, what didn't, what to try next. Action items.

→ Cadence: every sprint (2 weeks)
🔧

Code Health

Test coverage, flaky tests, tech debt markers, security scans.

→ Cadence: CI reports weekly
🔄 RETROSPECTIVE FORMAT
# Sprint 12 Retrospective (2026-04-15)

## Data Check-In (5 min)
- Velocity: 34 pts (avg 31)
- Incidents: 2 (1 SEV-3, 1 SEV-4)

## What Went Well

## What Didn't

- Code review took > 2 days on 3 PRs

## Experiments for Next Sprint

- [ ] Try "review within 4 business hours" SLA
KONSEP 3/3

DATA-DRIVEN
PRIORITIZATION

"Siapa yang paling keras bicara" bukan framework prioritization. Pakai Impact × Confidence / Effort (ICE) atau RICE. Pasangkan setiap backlog item dengan evidence dari ops data.

📊 PRIORITIZATION MATRIX (RICE)
# Backlog Prioritization — Sprint 13

Formula: score = (Reach × Impact × Confidence) / Effort

| Item                          | R    | I   | C    | E  | Score | Rank |
|-------------------------------|------|-----|------|----|-------|------|
| Fix DB connection leak (PR#78)| 2100 | 3   | 100% | 2  | 3150  | 1    |
| Add bulk endpoint load test   | 2100 | 2   | 90%  | 3  | 1260  | 2    |
| SLO burn-rate alerting v2     | 2100 | 2   | 80%  | 5  | 672   | 3    |
| Dashboard — top slow queries  | 2100 | 1   | 90%  | 2  | 945   | 4    |
| Slug analytics API (new feat) | 1200 | 2   | 60%  | 8  | 180   | 5    |
| Custom domain support         | 300  | 3   | 50%  | 13 | 35    | 6    |

Reach = weekly users affected
Impact = 0.25 / 0.5 / 1 / 2 / 3 (minimal→massive)
Confidence = %
Effort = person-weeks
📈 R — REACH

Pakai angka konkret dari analytics, bukan tebakan.

🎯 I — IMPACT

Link ke SLO atau user outcome yang bisa diukur.

🧠 C — CONFIDENCE

% berdasarkan evidence. Tebakan murni = 50%.

## North Star
Jadi internal service yang paling reliable (99.9% SLO) dan paling
cepat di-iterate (deploy < 5 min, MTTR < 15 min).

Success criteria: median PR merge time < 24h.

## 90-day (Features)
- [ ] Slug analytics API (top clicks, referrer breakdown)

- [ ] Admin dashboard read-only view

## Not-Doing List

💡 THE NOT-DOING LIST

Explicit "tidak akan kami kerjakan" sama pentingnya dengan "akan kami kerjakan". Menghindari scope creep dan alignment expectation stakeholder.

QUIZ • VALIDATE

TEST
PEMAHAMAN

Saat postmortem meeting, seorang engineer bilang: "Budi yang deploy tanpa review, itu penyebab incident." Apa tanggapan paling tepat dari lead?

A. Setuju dan catat Budi sebagai owner action item untuk training ulang
B. Reframe: "Mengapa sistem kita memungkinkan deploy tanpa review? Apa yang bisa kita ubah supaya ini tidak bisa terjadi lagi, terlepas siapa yang deploy?"
C. Minta Budi minta maaf di depan tim supaya jadi pelajaran
D. Pindahkan diskusi ke 1:1 dengan Budi — bukan urusan tim

Jawabannya B — blameless postmortem memindahkan fokus dari "manusia" ke "sistem". Kalau deploy tanpa review secara teknis mungkin, maka branch protection rule atau CI gate yang salah, bukan Budi. Reframe ke "bagaimana sistem memungkinkan ini" menghasilkan action items yang mencegah masalah kambuh, dan menciptakan psychological safety yang bikin orang melaporkan insiden (bukan menyembunyikannya).
