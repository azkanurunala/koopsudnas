KONSEP 1/3

INLINE EDIT
VS COMPOSER

Aturan sederhana: Cmd+K untuk 1 file, Cmd+I untuk banyak file. Keduanya punya feel yang berbeda dan butuh prompting yang berbeda.

CMD+K • INLINE EDIT

Single file, surgical

# Scope: selected lines / current cursor position
# Best for:
- Rename within function
- Add error handling to a block
- Convert callback to async/await
- Fix specific bug in selection
- Generate a single function from scratch

# Prompt style: short, imperative
"convert to async/await with try-catch"
"add zod validation for the input"
"extract this block into a helper function"
CMD+I • COMPOSER

Multi-file, agentic

# Scope: @file refs or @codebase, can create/delete files
# Best for:
- Add new feature across layers
- Rename symbol codebase-wide
- Migrate library version
- Generate tests for multiple modules
- Scaffold CRUD with routes + service + test

# Prompt style: detailed spec, list files
"Add a soft-delete feature. Update:
 @src/db/schema.ts  @src/services/user.ts
 @tests/user.test.ts  Add deletedAt column,
 filter queries, update tests."
🎯 WHEN TO SWITCH

Start dengan Cmd+K. Kalau kamu sadar task butuh > 1 file, langsung pindah ke Cmd+I. Jangan paksakan Cmd+K untuk refactor multi-file — hasilnya akan inkonsisten.

KONSEP 2/3

.CURSORRULES
TEAM CONVENTIONS

File .cursorrules di root project otomatis di-inject ke setiap prompt. Ini tempat kamu "ajarkan" Cursor tentang arsitektur, konvensi, dan larangan project.

EXAMPLE .cursorrules
# .cursorrules

You are an expert TypeScript + Next.js 14 (App Router) developer
working on a fintech product. Follow these rules STRICTLY:

## Stack
- TypeScript strict mode, no `any`, no `@ts-ignore`
- Next.js 14 App Router, Server Components by default
- Drizzle ORM for database (PostgreSQL)
- Zod for all runtime validation
- Tailwind CSS + shadcn/ui components

## Code Style
- Functions: max 40 lines, extract helpers freely
- Imports: use `@/` alias, group std → third-party → local
- Error handling: throw AppError subclasses, never silent catch
- No default exports except Next.js conventions

## Security
- NEVER log PII (name, email, phone, account numbers)
- NEVER inline user input into SQL/queries
- Always use transactions for multi-write operations
- All server actions must validate session

## Testing
- Vitest, one `*.test.ts` per module
- Cover: happy path, edge cases, error paths, at least 1 integration test

## Avoid
- `useEffect` if Server Component suffices
- Custom hooks for single-use logic
- Premature abstraction (rule of three applies)
✅ TIPS

Pakai .cursorrules untuk konvensi yang tidak boleh diubah. Update bersama tim via PR.

⚠️ PITFALLS

Jangan terlalu panjang (> 500 baris) — budget context habis untuk rules. Fokus ke yang penting.

KONSEP 3/3

CODEBASE
INDEXING & @REFS

Cursor mengindeks project kamu dengan embeddings. Symbol @ di chat/composer = semantic retrieval — hanya potongan relevant yang masuk prompt, bukan seluruh file.

@REFERENCES CHEAT SHEET
@file:src/services/order.ts       # entire file
@folder:src/services              # all files in folder
@symbol:processOrder              # jump to function def
@codebase                         # semantic search project-wide
@docs:Next.js                     # pull in framework docs
@git                              # recent commits as context
@web                              # live web search
@lint                             # current lint errors
@terminal                         # terminal output buffer
💡 INDEXING TIPS
  • • Indexing awal butuh waktu (repo 100k LOC ~5-10 menit)
  • • Tambah pattern di .cursorignore untuk skip vendor/dist
  • • Re-index setelah big refactor via settings
🎯 WHEN TO USE @
  • • @codebase untuk wide search/question
  • • @file: saat kamu tahu file exact
  • • @docs untuk API yang kamu tidak hafal

TASK: ADD PAGINATION TO USERS API
# Step 1: Open Composer (Cmd+I)

# Step 2: Prompt with file refs
@src/routes/users.ts @src/services/user-service.ts @src/db/queries/users.ts

Add cursor-based pagination to the GET /api/users endpoint:
1. Accept ?cursor= and ?limit= query params (limit max 100)
2. Service returns { items, nextCursor }
3. Query uses WHERE id > cursor ORDER BY id LIMIT :limit+1
4. Update route to return { data, pagination: { nextCursor } }
5. Add tests covering: no cursor, with cursor, end-of-list, limit cap

Don't touch auth middleware. Preserve existing error handling.

# Step 3: Composer shows proposed changes per file
#   → Review each diff, Accept All or cherry-pick

# Step 4: Run tests, iterate with follow-up
"The limit cap test is failing — the validation throws the wrong
error. Check @src/errors.ts and use ValidationError instead."
PROMPT

Explicit file refs + numbered requirements + "don't touch X".

REVIEW

Composer tunjukkan diff per file. Review sebelum accept.

ITERATE

DEEP PATTERNS

Tiga pattern Composer yang benar-benar bikin Cursor jadi "IDE yang refactor dirinya sendiri".

🔤

Codebase-Wide Rename Refactor

COMPOSER

Rename concept yang muncul di puluhan file dengan semantic understanding:

# Cursor Composer — rename concept
@codebase Rename the concept "Customer" to "Account" throughout
the codebase. Include:
- Type/interface names, DB schema, service functions, routes
- Database column `customer_id` → `account_id` (schema + migrations)
- Tests and fixtures
- Error messages and logs mentioning "customer"

Do NOT rename variables that refer to literal customer strings
in marketing copy. Generate a migration file for the DB change.
📦

Framework Version Migration

MIGRATE

Upgrade framework version dengan breaking changes:

# Cursor Composer — migrate Next 13 → 14
@codebase @docs:Next.js Migrate this Next.js 13 app to Next.js 14
App Router. Specifically:
1. Convert pages/ routes to app/ directory equivalents
2. Replace getServerSideProps with Server Components
3. Update next/navigation imports
4. Replace Image component `layout` prop with `fill` / sizing
5. Update middleware.ts for new matcher syntax

Preserve all business logic. Update package.json. Show me the
migration as a plan first, then execute file-by-file.
🧪

Batch Test Generation

TESTS

Generate tests untuk semua modul yang belum tercover:

# Cursor Composer — batch test gen
@folder:src/services For every *.ts file in this folder that does
NOT have a corresponding *.test.ts, generate a Vitest file that:

1. Covers all exported functions
2. Includes happy path + at least 2 edge cases + 1 error path
3. Mocks DB and external services properly
4. Uses existing test utilities from @src/test-utils.ts

Run `npm test` after and fix any failing ones.
🔑 Pro tip: Selalu minta Composer "plan first" untuk task besar. Review plan → approve → execute. Lebih aman daripada direct execute yang susah di-undo.
QUIZ • VALIDATE

TEST
PEMAHAMAN

Kamu perlu menambahkan field updatedBy ke 7 tabel dan semua query yang men-write ke tabel tersebut. Tool mana di Cursor yang paling tepat?

A. Cmd+K di setiap file satu per satu
B. Cmd+I Composer dengan @codebase dan spec detail, minta plan dulu
C. Chat panel dengan pertanyaan umum "add updatedBy"
D. Tab completion sambil mengetik manual

Jawabannya B — task cross-file yang span 7 tabel butuh Composer (Cmd+I). @codebase memastikan semantic context, plan-first memungkinkan kamu review sebelum execute. Cmd+K terlalu lokal, chat umum terlalu ambigu.

LATIHAN

📝 Task

Di project mana pun, tambahkan health check endpoint GET /api/health yang cek database + Redis + external service. Pakai Composer end-to-end.

# Steps

1. Setup .cursorrules minimal dengan stack project kamu
2. Open Composer (Cmd+I)
3. Prompt dengan spec + file refs yang relevan
4. Minta plan dulu, review, approve
5. Execute, run tests
6. Follow-up jika ada issue tanpa re-explain
🤖 STARTER PROMPT
@codebase Add a health check endpoint at GET /api/health:
- Check database connectivity (SELECT 1)
- Check Redis with PING
- Check external API with a 500ms-timeout HEAD request
- Return { status: "ok" | "degraded", checks: { db, redis, external } }
- 200 if all ok, 503 if any failed
- Add integration test with mocks

Plan first, then execute. Don't break existing routes.
