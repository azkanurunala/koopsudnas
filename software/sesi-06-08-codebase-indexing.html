KONSEP 1/3

EMBEDDINGS:
CODE → VECTOR

Embedding model mengkonversi text/code menjadi vektor high-dimensional (misal 1536 dimensi). Kode dengan makna mirip = vektor yang berdekatan secara cosine similarity.

// Compute embedding
const { embedding } = await openai.embeddings.create({
  model: "text-embedding-3-small",
  input: `
    function calculateTotal(items) {
      return items.reduce((sum, i) => sum + i.price, 0);
    }
  `
});

// embedding = [0.023, -0.041, 0.108, ..., 0.067]  // 1536 numbers

// Similarity (cosine)
function cosine(a, b) {
  const dot = a.reduce((s, v, i) => s + v * b[i], 0);
  const magA = Math.sqrt(a.reduce((s, v) => s + v*v, 0));
  const magB = Math.sqrt(b.reduce((s, v) => s + v*v, 0));
  return dot / (magA * magB);  // 1 = identical, 0 = unrelated
}
KONSEP 2/3

RAG PIPELINE
FOR CODE

Retrieval-Augmented Generation: index codebase sekali, retrieve relevant chunks per query, inject ke LLM context. Alternatif untuk codebase yang tidak muat long context.

# Stage 1: Indexing (once)
1. Walk codebase, read files
2. Chunk each file (by function/class, not fixed-size)
3. For each chunk: compute embedding
4. Store in vector DB: (chunk, embedding, metadata)

# Stage 2: Query (per user question)
1. User: "where is the auth middleware?"
2. Compute embedding of query
3. Vector DB: nearest neighbor search (top-K)
4. Retrieve K chunks (e.g., K=5)
5. Inject chunks into LLM prompt as context
6. LLM answers grounded in retrieved code
KONSEP 3/3

CHUNK
STRATEGY MATTERS

Fixed-size chunks (e.g., 500 tokens) merusak semantik. Untuk code, lebih baik chunk berdasarkan struktur AST: one function/class per chunk.

❌ Naive Chunking

// Fixed 500-char chunks
chunk1: "...ay = 5;\n  const y = 10;\nfunction calcul..."
chunk2: "...ateTotal(items) {\n  return items.reduce..."

# Function split across chunks!
# Embedding can't capture full meaning.

✅ AST-Based Chunking

// One function per chunk
chunk1: {
  path: "src/math.ts",
  code: "function calculateTotal(items) {\n ... }",
  symbol: "calculateTotal"
}

# Self-contained, meaningful embedding.
TRADEOFF

LONG CONTEXT VS
RETRIEVAL

Long Context (1M tokens)

✓ Full codebase visibility
✓ No chunking loss
✓ Cross-file reasoning
✗ Mahal per query
✗ Latency tinggi
✗ "Needle in haystack" issues

Best for: multi-file refactor, architectural questions, small-medium codebase.

Retrieval (RAG)

✓ Murah (chunks saja)
✓ Scale ke huge codebase
✓ Low latency
✗ Chunk boundaries
✗ Cross-file reasoning limited
✗ Retrieval quality = gating factor

Best for: "find where X is done", isolated function edits, huge codebase.

💡 Hybrid approach: RAG to retrieve, then feed to long-context model untuk reasoning. Best of both — Claude Code dan Cursor sudah pakai pattern ini.

📂

Index Large Repo — Tool Choice

TOOLS

Options untuk index large repo: Sourcegraph Cody, Continue.dev, Cursor built-in, custom pgvector/Qdrant.

# Cursor: built-in (automatic on open)
# Cody: cloud, secure + enterprise
# Continue.dev: open source, local
# Custom: pgvector + LangChain + tree-sitter

# Recommendation order (usually):
1. Cursor first (free, integrated)
2. Cody for team/enterprise
3. Custom only if you have unique constraints
🔎

Query with Retrieval

QUERY

Natural-language query ke Cursor @codebase atau Claude Code — let the tool retrieve relevant chunks.

# Cursor chat
@codebase Where is our rate limiting implemented?
Walk me through the middleware chain for an authed request.

# Claude Code
Trace how a POST /api/upload request flows through our codebase.
Include middleware order, validation, storage layer. Cite file:line.
🔀

Hybrid: Retrieval + Long Context

HYBRID

Retrieve broad → feed to long-context → reason deep.

# Claude Code dengan Opus 1M context
First use retrieval to find all files related to "payment processing":
- search for "payment", "charge", "stripe"
- include middleware, routes, services, tests

Then load these files into context and do a security audit:
- webhook signature validation
- idempotency keys
- error handling on 3rd party failures
- PII exposure in logs
⚠️ Pitfall: Retrieval quality sangat bergantung chunk boundaries. Untuk code, selalu pastikan tool pakai AST-aware chunking (function/class level), bukan fixed-size.
QUIZ

TEST
PEMAHAMAN

Kamu punya monorepo 2M+ LOC. Tim sering bertanya "di mana logic X diimplementasi?" Pendekatan paling scalable?

A. Pakai Claude Opus 1M context, load seluruh monorepo tiap query
B. RAG dengan AST-aware chunking (Cursor/Cody indexed)
C. grep + regex di CI
D. Manual documentation wiki

Jawaban B — 2M LOC tidak muat dalam 1M token context (rata-rata 4 chars/token → ~500K tokens bisa muat, sisanya harus retrieval). RAG dengan AST chunking + Cursor/Cody paling scalable. grep (C) bagus untuk literal search tapi miss semantic.
